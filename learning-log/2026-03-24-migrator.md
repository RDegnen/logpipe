# 2026-03-24 – DB Migrator

## What We Covered

### Why a Go binary instead of a bash script
- Bash + psql works but requires psql installed locally and feels out of place in a Go codebase.
- A Go migration library (goose, golang-migrate) would pull in a heavy dependency for ~50 lines of work.
- `cmd/migrator` using `pgx` directly is consistent with the rest of the project and teaches how migration tracking works under the hood.

### How idempotency works
- A `schema_migrations` table tracks which files have been applied (`name TEXT PRIMARY KEY`, `applied_at TIMESTAMPTZ DEFAULT now()`).
- On each run: create the table if not exists, then for each `.sql` file check if its name is already recorded — skip if yes, apply and record if no.
- `name` as primary key is the idempotency mechanism: inserting a duplicate name would fail, so we check first.

### Why ordering matters
- Migrations can build on each other (e.g. a later migration might add a column to a table created by an earlier one).
- Files are numbered with zero-padded prefixes (`001_`, `002_`) so lexicographic sort from `os.ReadDir` gives the correct application order.

### pgx.Connect vs pgxpool
- Short-lived CLI tools use `pgx.Connect` (single connection, closed on exit).
- Long-running services use `pgxpool.New` (connection pooling for concurrent queries).
- The migrator is a CLI tool, so `pgx.Connect` is the right choice.

### What was built
- `cmd/migrator/main.go` — Go binary that applies pending migrations idempotently.
- `Makefile` — added `migrate` target (`go run ./cmd/migrator`).
